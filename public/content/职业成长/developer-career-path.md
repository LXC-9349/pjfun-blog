## 这不是一篇路线图文章

网上太多程序员职业路线图了——第 1 年学什么，第 3-5 年做到什么级别。这些路线图的问题在于，它们把所有人和所有公司都当成一样的。

事实上，大厂的 P5-P8 路径在小公司不存在。创业公司的 CTO 发展路径在大厂也不适用。这篇文章不谈"你应该在哪个阶段做什么"，而是分享几个对我影响很大的**职业选择时刻**，以及每个选择背后的思考逻辑。

## 第一年：不要纠结技术栈

我第一份工作用 jQuery 写后台管理页面，当时觉得很丢人——别人的项目都用 Vue/React 了，我还在操作 DOM。有段时间非常焦虑，下班自学 React。

现在回头看，那一年对我帮助最大的恰恰不是新技术——而是学会了怎么把一个页面需求拆成可实现的步骤、怎么和产品经理沟通"这个做不了"、怎么在三天内学会一个完全没接触过的库并用到生产环境。

技术栈会变，但 **"把一个需求变成代码"的能力不会变**。新手阶段最重要的不是学了什么框架，而是建立"交付闭环"——需求→设计→编码→自测→上线，完整走几轮。

那位当年带我的人说了一句话我一直记得：**"前三年你不需要选方向，你只需要积累完成度。"** 完成度就是"答应的事情一定能做成，而且质量不会太差"。这个信誉一旦建立起来，后面很多机会会自动找到你。

## 第三年：第一次选方向

三年左右你不再是一个完全的初学者了。这时候要开始选择深耕的方向——但大多数人选错了方式。

### 什么不是选方向

- "前端有前景吗？""后端是不是天花板更低？"——这是选赛道，不是选方向
- "Node.js 好像不错的"——这是选工具，不是选方向

### 什么是选方向

选方向是选择一个**问题领域**。比如：
- 我选择成为"前端性能专家"，所有项目我都关注加载速度、运行时性能、缓存策略
- 我选择成为"数据基础设施工程师"，所有项目我都关注数据流、存储、查询效率、一致性保证

我选了后者。并不是因为我觉得前端性能没前途，而是因为我对"数据在系统中如何流转"这件事本身有好奇心。这种好奇心支撑了我在这个方向上持续投入。

关键：**选你真正有好奇心的事**。不是因为薪资高、不是因为热门——是因为你遇到这类问题时不会觉得烦。职业发展是长跑，没有好奇心支撑，跑不下去。

## 第五年：技术深度的陷阱

第五年的时候，我已经能在自己的领域解决大部分技术问题了。这带来一种虚幻的安全感——"我技术很牛"。

但问题在于：**技术能力有边际效应递减**。从 80 分到 90 分需要花的精力，可能比从 30 分到 80 分还多。而 90 分到 95 分需要花的精力更多，但业务可能根本感受不到区别。

这时候我遇到了一件让我反思的事：一个"技术比我差"的同事，在公司的影响力比我大得多。他做了很多不在代码里的事——推动跨团队协作、写设计文档让所有人对齐、在项目初期就识别出风险。

这个阶段的关键转变是：**从"我多强"到"团队多强"**。

你的价值不再是"你能写出多难的技术方案"，而是"你能让团队交付多少价值"。这包括指导初级工程师、建立代码规范、推动技术决策、和产品沟通排期。

这不是说技术不重要——而是技术只是价值的一部分。单维度比拼技术能力，到一定阶段后收益会越来越低。

## 第七年：架构思维的形成

架构能力不是"学"出来的，是"养"出来的。它是长期面对复杂系统后沉淀下来的一种感觉。

我对自己感到明显变化的是第七年左右：面对一个新需求时，脑子里不再是一个"怎么实现"的路径，而是会同时出现三四种方案，并且能快速评估每种方案的代价和收益。

```python
# 这不是代码，是思考方式的变化

# Before：拿到需求→想一个实现方案→开始写
def approach_before(requirement):
    solution = design_one_solution(requirement)
    code(solution)

# After：拿到需求→列出多种方案→横向对比→选一个
def approach_after(requirement):
    candidates = [
        solution_a(requirement),  # 简单但扩展性差
        solution_b(requirement),  # 复杂但灵活
        solution_c(requirement),  # 介于两者之间
    ]
    trade_offs = analyze(candidates)
    chosen = select_based_on_context(trade_offs)
    return chosen
```

这个变化不是看了某本书或某个课程带来的——是在踩了足够多的坑之后自然形成的。每次做错的技术决策，每次因为设计不合理而重构，都在积累这种"感觉"。

给还没到这个阶段的人的建议：**多复盘**。每次上线后，不管有没有出问题，写一段总结——这个方案哪里做得好，哪里不好，下次遇到类似情况会怎么做。三年前的复盘笔记，现在看可能觉得幼稚——但那就对了，说明你在成长。

## 技术管理：不是升职，是转行

很多人把"做管理"当成升职，这是一个误解。从工程师到技术经理，本质上是在**换一个职业**——就像从建筑师转行做项目经理一样。

管理需要的能力包括：
- 拆解目标：把大目标拆成分步、可衡量的小目标，分配到人
- 知人善用：判断团队里每个人的优势和短板，分配合适的任务
- 决策：在不完全信息下做决定，并为结果负责
- 沟通：对上争取资源，对下传递方向，对平级协作
- 兜底：出了问题不是你解决了就行——而是团队里没有人解决时，你来顶

这些能力和写代码需要的能力（逻辑严密、注意力集中、完美主义）在很多方面是冲突的。

有一件事让我决定不做纯管理：前主管和我谈话时说，"你最开心的日子是你写代码的那几天，不是上线后、不是绩效评估时——是你自己写出了一段好代码的那天。" 他说得对。我从中获得的满足感，和推动团队达成目标获得的满足感，是两回事。我不需要一定选一个，但至少要清楚自己在哪边获得更多能量。

## 写代码还是写文档？

写代码和写文档不是二选一。那些对你职业发展帮助最大的人，通常是**代码写得好、文档也写得清楚**的人。

文档的价值：
- 设计文档是"把思考过程公开"——别人可以看到你的决策逻辑，参与讨论，避免你一个人想偏
- 记录文档是"帮未来的自己减负"——三个月后你回来看这段代码，如果没有注释和文档，你可能也想不起来当初为什么这么写

推荐一个习惯：**写 RFC 文档**。即使团队没有强制要求，你也可以在做一个重要决定之前，写一页 A4 纸的 RFC（Request For Comments），发给相关同事。这不仅帮别人理解你的方案，更重要的是逼自己在写的过程中想清楚。

## 写在最后

职业发展不是打怪升级。没有一条"正确"的路——只有适合你的路。

几个对我有帮助的长期原则：

1. **做有积累的事**：选一个方向，持续投入，让经验可以叠加。不要今年搞前端、明年搞大数据、后年搞 AI

2. **保持可迁移性**：技术的选择要考虑"如果我换一家公司，这个能力还能用吗？"

3. **关注影响力边界**：你的实际影响力 = 职责 × 能力 × 可见度。三者缺一不可

4. **定期清空**：每 6 个月到 1 年，审视一下自己的知识体系——哪些已经过时了，需要更新了

5. **善待同事**：技术圈子不大。你在一个地方的人品，会跟着你到下一个地方
