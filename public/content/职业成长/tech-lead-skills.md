## Tech Lead 和 Senior Engineer 的区别

很多人以为 Tech Lead 就是"写代码最厉害的人"。这是最大的误解。

| 维度 | Senior Engineer | Tech Lead |
|------|----------------|-----------|
| 关注点 | 自己的代码质量 | 团队的整体产出 |
| 时间分配 | 80% 编码，20% 其他 | 40% 编码，60% 其他 |
| 成功标准 | 自己完成了多少 | 团队完成了多少 |
| 决策范围 | 技术方案 | 技术方向 + 人员安排 |
| 沟通对象 | 同事 | 同事 + 产品 + 管理层 |

**核心转变**：从"我自己能做好"到"让团队能做好"。

## 技术决策能力

### 技术选型评估矩阵

当需要在多个方案中选择时，用评估矩阵：

```
需求：选择前端框架

评估维度          权重    React    Vue    Svelte
学习曲线          15%      7        9       8
生态成熟度        25%      10       8       5
团队熟悉度        20%      8        6       3
性能              15%      7        8       9
长期维护          15%      9        7       6
招聘难度          10%      9        8       5
─────────────────────────────────────────────
加权得分          100%    8.25     7.55    6.15
```

**决策原则**：
1. 列出所有评估维度
2. 给每个维度分配权重（总和 100%）
3. 给每个方案打分（1-10）
4. 计算加权得分
5. **但不要被数字绑架**——数字是辅助，不是决定

### 技术债务管理

```
技术债务四象限：

高价值 ┃ 高价值
低债务 ┃ 高债务
       ┃
───────╋───────
       ┃
低价值 ┃ 低价值
低债务 ┃ 高债务

策略：
左上：立即做（高价值、低债务）
右上：计划做（高价值、高债务）
左下：考虑做（低价值、低债务）
右下：不做（低价值、高债务）
```

**什么时候该还技术债务**：
- 债务导致 bug 频发
- 债务阻碍新功能开发
- 债务让新人 onboarding 困难
- 团队有 20% 的时间可以分配

**什么时候可以欠**：
- 验证一个假设（MVP）
- 赶一个不可错过的 deadline
- 但这个功能可能很快被废弃

> **规则**：欠债可以，但要记录在案，并计划偿还时间。

## 代码审查的艺术

### 建设性反馈 vs 挑刺

```
❌ 挑刺：
"这个变量名不好，改成 x"
"你为什么不用 hook？"
"这段代码有性能问题"

✅ 建设性反馈：
"这个变量名 `tempData` 不太能表达它的用途，
  改成 `userProfileCache` 会不会更清晰？"

"这里用 useEffect 可能会导致多次渲染，
  考虑用 useMemo 缓存结果？"

"这个循环在数据量大时可能成为瓶颈，
  建议用 Map 替代数组查找，复杂度从 O(n) 降到 O(1)"
```

### 代码审查清单

- [ ] 功能是否正确？
- [ ] 是否有安全隐患？（SQL 注入、XSS、权限）
- [ ] 是否有边界情况没处理？
- [ ] 代码是否可读？
- [ ] 是否有不必要的复杂度？
- [ ] 测试是否覆盖了这个改动？
- [ ] 是否符合项目的编码规范？

### 审查原则

1. **先理解意图，再看实现**——不要一上来就挑毛病
2. **区分"必须改"和"建议改"**——用 "nit:" 标记非必须的建议
3. **一次不要提太多**——超过 10 条评论会让作者崩溃
4. **当面沟通复杂问题**——不要在 PR 里来回辩论 20 轮

## 任务分解和估算

### 如何把大需求拆成小任务

```
需求：实现用户积分系统

❌ 错误分解：
- 做积分系统（5 天）

✅ 正确分解：
- 设计积分数据模型（0.5 天）
- 实现积分增减 API（1 天）
- 实现积分查询 API（0.5 天）
- 前端积分展示（1 天）
- 积分变动通知（0.5 天）
- 积分排行榜（1 天）
- 测试和文档（0.5 天）
总计：5.5 天
```

### 估算技巧

- **三点估算**：乐观 + 最可能 + 悲观，取加权平均 `(O + 4M + P) / 6`
- **类比估算**：和之前做过的类似功能对比
- **分解估算**：拆到每个子任务不超过 1 天
- **留 buffer**：估算 × 1.5（未知因素总是存在）

> **经验**：开发者的估算通常偏乐观。作为 Tech Lead，你需要帮他们加上 buffer。

## 团队沟通

### 向上管理

```
❌ 错误汇报：
"我们在重构代码，还需要一周"

✅ 正确汇报：
"我们正在优化订单模块的性能，
  完成后页面加载时间预计从 3s 降到 1s。
  目前进度 70%，预计下周三完成。
  风险：如果第三方支付接口有变动，可能延后 2 天。"
```

**向上管理的关键**：
- 说业务价值，不说技术细节
- 给进度，给风险，给方案
- 不要等出了问题才汇报

### 向下指导

```
❌ 直接给答案：
"这里用 reduce 就行了"

✅ 引导思考：
"你觉得这段代码的目的是什么？
  有没有更简洁的方式来实现？
  想想数组的哪些方法可以帮到我们？"
```

**指导原则**：
- 教方法，不给答案
- 允许犯错，事后复盘
- 根据经验水平调整指导深度

## 个人时间管理

### 编码时间 vs 管理时间

```
理想的 Tech Lead 时间分配：

编码     ████████░░░░░░░░░░░░  40%
会议     ████░░░░░░░░░░░░░░░░  20%
Code Review ███░░░░░░░░░░░░░░  15%
指导     ██░░░░░░░░░░░░░░░░░░  10%
规划     ██░░░░░░░░░░░░░░░░░░  10%
其他     █░░░░░░░░░░░░░░░░░░░   5%
```

### 保护编码时间

- **设定"免打扰"时段**：每天 2-3 小时不参加会议
- **选择影响小但重要的任务**：不要碰核心路径，选工具链、文档、技术债
- **结对编程**：在指导的同时写代码

## 常见陷阱

### 微观管理

```
❌ 微观管理：
"这个函数要这样写..."
"你为什么用 for 循环不用 map？"
"每行代码都要经过我审核"

✅ 授权：
"这个模块你负责，方案你定，
  有分歧我们再讨论。"
```

### 技术独裁

```
❌ "必须用我选的技术栈"
✅ "我们评估一下几个方案的优劣"

❌ "我的代码不需要 review"
✅ "我的代码也欢迎大家提意见"
```

### 过度授权

```
❌ "这个你看着办"（然后完全不跟进）
✅ "这个你负责，周五我们过一下进度，
    有什么困难随时找我"
```

## 招聘和面试

### 如何识别优秀人才

**看这些特质**：
- 好奇心：是否主动学习新技术
- 沟通能力：能否清晰表达想法
- 责任心：是否对自己的代码负责
- 协作精神：是否愿意帮助他人

**不要只看**：
- 背了多少八股文
- 会不会手写红黑树
- 在多少家公司待过

### 面试建议

```
好的面试流程：
1. 技术面（60 分钟）：实际编码，不是背题
2. 系统设计（45 分钟）：开放性问题，看思考过程
3. 行为面（30 分钟）：过去的项目经验
4. 团队面（30 分钟）：文化匹配度
```

## 总结

Tech Lead 的核心能力模型：

```
         技术深度
           ▲
          / \
         /   \
        /     \
  沟通能力 ─── 决策能力
        \     /
         \   /
          \ /
           ▼
        团队管理
```

四个维度缺一不可。技术深度让你有话语权，沟通能力让你能传达想法，决策能力让你选对方向，团队管理让你能放大影响力。

记住：**Tech Lead 的成功不是自己写了多少代码，而是团队交付了多少价值。**
