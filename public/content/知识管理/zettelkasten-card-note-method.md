## 为什么会接触卡片笔记

事情要从两年前说起。我在学 React 源码，每天看很多文章和源码分析，做了一堆笔记。三个月后翻看，发现大多数笔记都只是"某篇文章的摘要"——我把别人的理解抄了一遍，不是自己的思考。

然后我读到了《How to Take Smart Notes》这本书。里面的方法和我的困境高度重合：学了很多，但知识之间没有连接，无法形成体系。

于是我开始尝试 Zettelkasten（卡片笔记法）。到现在两年多了，笔记库积累了 800 多张卡片。这篇文章不讲理论（书里都有），讲**我在学编程时具体是怎么用的**。

## 三类笔记

### 闪念笔记（Fleeting Notes）

这是最轻量的一种。看到一个有意思的推文、群里讨论的一个问题、读文章时闪过的一个念头——随手记下来。

```markdown
# 2026-05-16 想法
React 的 useTransition 其实相当于一个"状态降级"机制——
让低优先级的更新可以被高优先级更新打断。
这和浏览器的事件循环机制有点像：微任务优先于宏任务。
```

这种笔记不需要完整、不需要工整。它的作用是在**想法还在的时候抓下来**。人的工作记忆只有 4 个左右的槽位——不记下来就会被挤掉。

我放在 Obsidian 的 Daily Note 里，每天一个文件。周末整理时过一遍，有价值的提炼成永久笔记，没用的删掉。

### 文献笔记（Literature Notes）

读文章、看书、看视频时做的笔记。关键是——**不是摘抄，是用自己的话总结**。

```markdown
# 文献笔记：Vue 3 响应式原理（来源: vuejs.org/guide/extras/reactivity）

Vue 3 使用 Proxy 替代了 Vue 2 的 Object.defineProperty。
区别：
1. Proxy 可以代理整个对象，不需要递归遍历属性
2. Proxy 可以检测到属性的新增和删除
3. 但 Proxy 是 ES6 特性，不支持 IE

Vue 3 的响应式系统分为三个角色：
- 响应式数据（reactive/ref）
- 副作用（effect/watchEffect）
- 追踪器（track/trigger）

关键路径：读取属性 → track() 收集依赖 → 修改属性 → trigger() 执行副作用

和我之前的理解不同：我一直以为 ref 是通过 Object.defineProperty 实现的，
实际上 ref 在 .value 的 getter/setter 中调用了 track/trigger，
底层还是依赖 reactive（通过 toReactive 转换）
```

对比摘抄式笔记："Vue 3 使用了 Proxy 而不是 Object.defineProperty"——这句话只是复制粘贴，不是理解。**用自己的话写一遍，才是思考的过程**。

### 永久笔记（Permanent Notes）

这是核心资产。每条笔记只包含**一个概念**，用完整的句子写清楚，并且与其他笔记建立链接。

```markdown
# 响应式系统的本质是发布-订阅模式

在 Vue、React、Svelte 等框架中，响应式系统的核心机制都是
**发布-订阅模式（Pub/Sub）**的变体。

有三个角色：
1. 发布者（数据源）：数据发生变化时通知订阅者
2. 订阅者（副作用/渲染函数）：当数据变化时重新执行
3. 调度器：决定什么时候执行订阅者（同步、微任务、异步）

不同框架的实现差异：
- Vue 3：粒度细（组件级+节点级），依赖自动追踪
- React：粒度粗（组件级），手动声明依赖（useEffect deps）
- Svelte：编译时静态分析依赖

**延伸：** [[虚拟 DOM 的作用之一是降低响应式系统的复杂度]]
**延伸：** [[信号（Signal）机制是响应式系统的另一种实现]]
**延伸：** [[Vue 3 的响应式系统原理]]（文献笔记）
```

永久笔记的关键：
1. **原子性**：一条笔记只讲一个概念
2. **完整句**：即使脱离上下文也能理解
3. **链接**：和其他笔记建立关系

## 在编程学习中怎么用

### 场景一：学习新技术

假设你在学 Docker。传统的学习方式：看完一本书或一门课程，然后开始用。问题是你可能很快就忘了细节。

卡片笔记的方式：

```
先从几个核心概念切入：镜像 vs 容器、数据卷、网络模式。
每个概念写一张永久笔记。
读的过程中遇到具体操作，写成文献笔记。
发现概念之间有联系时，建立链接。
```

比如：

```markdown
# 镜像和容器的关系类似于类和实例
- 镜像是模板（静态），容器是运行实例（动态）
- 一个镜像可以启动多个容器
- 容器可以被启动、停止、删除，不影响镜像
- 对容器的修改不会影响镜像（除非 commit）

**延伸：** [[Docker 的 Union File System 让镜像分层成为可能]]
**延伸：** [[容器和虚拟机的本质区别在于共享宿主操作系统内核]]
```

这样你需要的不是"记住 Docker 的所有命令"——你需要的是理解这些核心概念，然后命令自然就知道在哪里用了。

### 场景二：调试 Bug

遇到一个 Bug，花了两个小时找到了原因和解决方案。大多数人修完就结束了。但如果你有笔记系统，这是一个很好的笔记素材。

```markdown
# Next.js 中 getStaticProps 的缓存行为

场景：在 getStaticProps 中调用了外部 API，但更新后页面没有刷新。
原因：getStaticProps 在开发模式下每次请求都会调用，
但在生产构建后，结果被序列化为 JSON 文件缓存。
只有重新构建才能更新数据。

解决方案：
1. 使用 fallback: 'blocking' + revalidate（ISR）
2. 对于实时数据，改用 getServerSideProps
3. 使用增量静态再生成

**延伸：** [[SSG 和 SSR 的选择标准]]
**延伸：** [[Next.js 的数据获取方法对比]]
```

下次遇到类似的问题，不是去 Google 搜"Next.js getStaticProps 缓存"——先翻翻自己的笔记库看看有没有相关记录。如果没有，就写一张新卡片。

久而久之，你自己的文档库变成了最贴近你实际项目的参考手册。

## 笔记之间的关系比笔记本身更重要

800 张笔记中的每一张单独看都没太大价值。有价值的是它们之间的连接。

```markdown
- [[响应式系统的本质是发布-订阅模式]]
  - [[Vue 3 的响应式系统原理]]
  - [[React setState 的批处理机制]]
  - [[Signal 机制]]
  
- [[React setState 的批处理机制]]
  - [[React 18 的自动批处理]]
  - [[flushSync 的使用场景]]
  - [[transition 更新的优先级]]
```

从一个节点出发，你可以沿着链接探索整个网络。这种"漫步"经常带来新的发现——你发现两个看似不相关的知识点之间其实有共同点。

有了链接之后还带来了一个好处：**上下文感**。当你看到一张旧笔记时，通过链接可以看到你当时在思考什么、和哪些概念做了对比。这比单纯的搜索关键词更有价值。

## 笔记库不是越大越好

两年 800 张卡片，平均每天一张多一点。为什么不每天记十张？因为**笔记的质量比数量重要**。

我的过滤原则：
1. 不记已经烂熟于心的内容
2. 不记过几天就会过时的信息（版本更新的 API 变化等）
3. 不记只是"记住"而没有"理解"的内容
4. 定期清理——如果一张笔记半年内没被引用过，考虑归档或删除

## 用什么工具

我用 Obsidian，因为：
- 纯 Markdown 文件，没有 vendor lock
- 双链和 Graph View 直观
- 插件生态丰富

但工具不是关键。用任何支持链接和搜索的笔记工具都可以——关键是用**写笔记的方式去思考**，而不是把笔记当作资料的搬运。
