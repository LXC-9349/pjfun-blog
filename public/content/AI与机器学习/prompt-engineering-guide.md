## 提示词不是咒语

接触 AI 一年多，看过很多人把写 Prompt 当成念咒——"请用专业且友好的语气，结合您的丰富知识，详细地..." 写了一大堆套话，结果输出还是不对。

Prompt Engineering 没那么玄乎。它本质上是在跟一个"很聪明但没有常识的系统"沟通。你需要把那些人类默认能理解的东西显式告诉它。

## 四条铁律

### 1. 指令要具体，不要模糊

```
# 差
写一篇关于人工智能的文章

# 好
写一篇1500字左右的中文文章，主题是"AI在医疗影像诊断中的应用"，
面向读者是非技术背景的医疗行业从业者，
需要包含3个真实案例，结论部分给出风险提示
```

区别在哪？后者定义了：受众、长度、结构、内容要求、调性。模型不需要猜测你想要什么。

### 2. 给格式约束，不给发挥空间

```
# 差
请列出 Python 的优缺点

# 好
请用以下格式列出 Python 的5个优点和3个缺点：

## 优点
### [优点标题]
[一句话说明]

## 缺点  
### [缺点标题]
[一句话说明]
```

这不是控制狂——大语言模型在格式约束下输出质量更稳定。模糊的指令会导致奇怪的输出结构。

### 3. 示例胜过描述

```
# 差
用正式的语气写一段关于远程协作的说明

# 好
请参考以下示例的语气，写一段关于远程协作的说明：

示例：
"项目管理不是监控每个人的工时，而是确保团队在正确的方向上协同前进。"

说明：
""
```

特别是用 One-shot 或 Few-shot 的方式给例子，效果比任何文字描述都好。这叫做"上下文学习"——模型从示例中推断你的期望。

### 4. 分步骤，不要一步到位

```
# 差
帮我写一份项目计划

# 好
第一步：列出这个项目的核心目标，不超过3条
第二步：基于目标，列出需要完成的里程碑
第三步：对每个里程碑，分解出具体任务
第四步：估算每个任务的时间和人力
```

分步指令利用了 Chain-of-Thought（思维链）的原理——把复杂问题拆成小步骤，模型推理过程更清晰，结果也更可靠。

## 角色扮演不是花活

很多人觉得给 AI 设定角色是在玩 cosplay，实际上角色扮演是一种有效的约束手段。

```
你是一名有10年经验的前端架构师，
负责评审团队的技术方案。
你的评审风格是：指出风险但不否定创意，
给出替代方案而不是只抱怨问题。
现在请评审以下方案：
[方案描述]
```

加了角色设定后，输出的口吻和视角会稳定——它知道"自己是谁"了。同一个问题，不加角色和加了角色，结果可能差很多。

但注意：角色的设定要**具体**。"专家"太泛，"有10年前端经验的架构师"才有效。

## 工作中的 Prompt 模板

### 代码审查

```
你是一个代码审查者。请审查以下代码，从以下维度给出反馈：
1. 安全隐患
2. 性能问题
3. 可维护性
4. 边界情况处理

代码：
```javascript
（贴代码）
``` 

请用以下格式：
### [严重级别] [问题类型]
[具体说明位置和问题]
[建议的修改方式]
```

### 技术方案讨论

```
我需要评估两个技术方案，请帮我做一个对比分析：

方案A：[描述]
方案B：[描述]

评估维度：
1. 实现复杂度（开发时间和难度）
2. 运行性能
3. 可维护性
4. 扩展性
5. 社区生态

请给出一个推荐方案并说明理由。
```

### 写周报

```
帮我写一份周报，包含以下信息：

本周完成：
- [完成的事项列表]
下周计划：
- [计划的事项列表]
遇到的问题：
- [问题列表]

要求：
- 每个事项写成一个bullet point
- 每个点包含：做了什么、结果如何
- 放在"团队同步"的上下文，不需要太正式
- 控制在200字以内
```

## 踩过的坑

### 坑一：上下文太长，模型会"失忆"

跟 AI 对话超过一定轮次后，模型会"忘记"上下文。不是因为 AI 不聪明，是因为注意力机制的窗口长度限制。

解法：**关键信息放前面和后面**。如果必须聊很多轮，定期把关键信息重新说一遍。

### 坑二：否定式指令可能起反效果

```
# 不推荐
不要写太长，不要用专业术语，不要用复杂句式

# 推荐
控制在200字以内，面向非技术读者，使用简单的句子
```

因为模型处理的是"长、专业术语、复杂句式"这些概念——否定并不能阻止它想起这些词。实验结果也表明，正面指令比否定指令效果好。

### 坑三：过于礼貌

跟 AI 不需要客气。每次对话加"请"、"谢谢"、"非常感谢"——这些 token 占用了上下文长度，但 AI 并不需要情感维护。写简洁的指令就够了。

### 坑四：忽略输出格式的容错性

```json
请返回 JSON 格式：{"name": "...", "age": ...}
```

模型可能会输出带 markdown 代码块格式的 JSON，或者包含额外注释。如果下游程序解析，需要做好容错：

- 如果输出被 ```json ... ```包裹，提取中间部分
- 如果有多余逗号，修复 JSON
- 如果字段名不完全匹配，做模糊匹配

## 高级技巧：System Prompt 的结构化

如果你在开发 AI 应用，system prompt 可以按这个结构组织：

```
## 角色
[你是谁]

## 任务  
[你要做什么]

## 上下文
[背景信息]

## 规则
[约束条件，3-5条，用编号列表]

## 输出格式
[期望的输出结构]

## 示例（可选）
[一个具体例子]
```

这种结构化的 system prompt 比一大段文字更清晰，也更容易维护和迭代。

## 说两句实话

Prompt Engineering 不是魔法。它不能把差模型变好，只是把好模型用好。如果模型本身能力不行，再好的提示词也白搭。

另外就是不要过度优化。很多人在 prompt 上花的时间远超收益。一个够用的 prompt 和完美的 prompt 在实际使用中可能只差 5%——这 5% 不值得花 200% 的时间去追。

最后，**测试比设计更重要**。同一个 prompt 在不同模型上表现不同，甚至同一个模型的不同版本也有差异。最好的方法不是读一百篇教程——而是写十个版本的 prompt，逐一测试，选最好的。
